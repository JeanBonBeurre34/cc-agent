A compact C2 agent for EDR/AMSI resilience testing.

cc-agent is a small, opinionated C2 agent intended for controlled environments. It is built to emulate modern attacker tradecraft: configurable User-Agent, randomized & customizable beacon intervals, and behavior designed to stress-test endpoint defenses, AMSI-backed inspection and EDR.

Platform: Windows (agent)
Focus: C2 & telemetry
Use case: Red / blue teaming
View on GitHub ↗ Quickstart
Designed for research & training only. Never deploy on systems you do not own or have explicit authorization to test.
The project support evasion techniques (including AMSI-aware behavior) to help evaluate defensive coverage – it is not guaranteed to bypass any specific EDR.
Example agent configuration
# Beacon & C2
[agent]
server      = "https://c2.example.com"
min_beacon  = 10      # seconds
max_beacon  = 45      # seconds

# HTTP profile
http_user_agent = "Mozilla/5.0 (Windows NT 10.0; Win64; x64)"

# Telemetry
tls_verify  = true
Core Features

cc-agent focuses on realism and control: you can shape how the agent looks on the wire, how often it calls home, and how it behaves under EDR & AMSI monitoring, while also offering a set of built-in actions to simulate post-exploitation workflows.

EDR / AMSI-oriented behavior

The agent is structured to emulate techniques used to reduce visibility under EDR and AMSI inspection: in-memory operation, staged behavior, and low-noise network activity. This helps you validate how well your stack surfaces and correlates suspicious activity.

EDR lab testing AMSI-aware flows Telemetry exercises

Configurable HTTP fingerprint

Tune the agent’s User-Agent and HTTP profile so it blends into realistic traffic patterns during exercises. This is useful for simulating different threat profiles and testing proxy, IDS/IPS, and EDR correlation rules.

Custom User-Agent Proxy-friendly Traffic profiling

Randomized & custom beaconing

Define minimum and maximum beacon intervals and let the agent pick randomized values inside that window, or pin a fixed interval for deterministic runs. This mirrors real-world C2 beacons and allows defenders to test their detection pipelines at different noise levels.

Randomized intervals Min/max control Detection tuning

Built-in operator actions

Beyond beaconing, cc-agent ships with a collection of pre-built commands to run common operator workflows: a simple SOCKS5-style pivot, file download from the host where the agent is running, screenshot capture, remote PowerShell execution, file upload from a remote HTTP server, reverse shell and many more built-in features.

Configuration

cc-agent is designed to be controlled primarily via configuration, so you can quickly adapt the same binary to different scenarios: stealthy long-range beacons, noisy detection drills, or tightly profiled lab tests.

Beacon behavior

Beaconing is fully configurable:

  • Custom min/max intervals in seconds.
  • Random jitter between those bounds.
  • Ability to simulate high-noise or low-noise operators.

In long-running exercises, you can vary the beacon profile to mimic different threat actors and study how that affects your alerting and triage workflows.

User-Agent & HTTP profile

The http_user_agent setting lets you control how the agent presents itself to proxies and inspection points:

  • Impersonate common browsers or system processes.
  • Test how well your stack flags off-profile UAs.
  • Run A/B experiments on detection logic.

This is ideal for red/blue team collaboration: defenders can write rules based on these profiles, then validate them against the agent.

1. Clone & build (Go)

git clone https://github.com/JeanBonBeurre34/cc-agent.git
cd cc-agent

# Build the agent (example)
# update with the actual build command from the repo
go build -o go-agent.exe main.go

Replace the build command with whatever is documented in the repository (README.md, Makefile, etc.).

2. Build with Docker (build.sh)

# From the repository root
# Ensure Docker is installed and running

# Make the helper script executable
chmod +x build.sh

# Run the Docker-based build
./build.sh

# The Windows executable (e.g. myServiceAgent.exe)
# will be copied to the current folder.

The build.sh script wraps the Docker build and extraction steps described in the project README.md, making it easier to produce the Windows agent from a Unix-like system.
